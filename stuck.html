1
Understand — 3 min
Rephrase. Example. Edge cases. What is input, what is output.

Constraints → complexity → pattern. What does n tell you? What does the ask tell you?

The 5 questions. Brute force first. What can be discarded? Trace small example.

Code — 10-15 min
Skeleton first. One concern at a time. No new thinking while typing.

12 universal cases. Pattern-specific cases. State complexity.
Total for Medium: ~25 min. Total for Hard: ~40 min. These are real targets — not 10 min. Anyone claiming 10 min on a genuinely new problem is lying or has seen it before.
The one rule that governs everything
Never code to think. Only code to transcribe.
By the time your fingers touch the keyboard, your brain should already know the full solution. If you're figuring things out while typing — stop. Go back to Phase 3. The moment you merge thinking and coding is the moment bugs multiply and time leaks.
Phase 1 — Understand (3 min, before touching keyboard)
Q1
What is the exact input and output?
Not vaguely — precisely. "Input is int[] of size n, values can be negative. Output is a single int." Write it down. Most wrong submissions come from misreading.
Q2
Rephrase in one sentence in your own words
Not the problem's words. Yours. "Find the contiguous slice of array with the largest total." If you can't do this — you haven't understood the problem. Read again.
Q3
Trace the given example by hand
Input → output manually. Every step. Don't skip. This builds the concrete mental model that abstract reading misses. If your trace doesn't match expected output — your understanding is wrong.
Q4
List edge cases before solving
Empty input. Single element. All same. All negative. Answer is entire array. Answer is single element. Sorted. Reverse sorted. Write these down now — you'll use them in Phase 5.
Q5
What are the constraints?
n = ? Values range? Time limit? These aren't just numbers — they're the pattern unlock. Read Phase 2.
Phase 2 — Pattern (2 min)
S1
Constraint → required complexity
n ≤ 20 → O(2ⁿ) → backtracking / bitmask DP
n ≤ 300 → O(n³) → interval DP, Floyd-Warshall
n ≤ 3000 → O(n²) → 2D DP, two loops
n ≤ 10⁵ → O(n log n) → sort, binary search, BFS/DFS
n ≤ 10⁶ → O(n) → sliding window, two pointers, hashmap
n ≤ 10⁹ → O(log n) → binary search on answer only
S2
Ask → pattern mapping
"Number of ways / count" → DP
"Minimum / maximum value" → DP, greedy, binary search on answer
"Shortest path / minimum steps" → BFS (unweighted), Dijkstra (weighted)
"All possible / generate all" → backtracking
"Top K / K closest" → heap
"Subarray / substring + max or min" → sliding window, prefix sum
"Feasibility — can you achieve X" → binary search on answer
S3
Input structure → pattern
Sorted array → binary search or two pointers
Tree → DFS recursion almost always
Graph → BFS shortest / DFS explore
Intervals → sort by start + sweep
Two sequences → 2D DP
Grid → BFS shortest path / 2D DP count paths
S4
Write your hypothesis
"I think this is [pattern] because [constraint/keyword]." Commit to it. A wrong hypothesis you can disprove is faster than open-ended searching. If hypothesis fails Phase 3 — come back here and pick the next candidate.
Phase 3 — Key Insight (5 min) — the hardest phase
Don't try templates. Interrogate the problem. These 5 questions extract the insight from the problem itself — not from your memory.
Q1
What makes this hard? Name the bottleneck.
State brute force in one sentence. "For every pair i,j check condition — O(n²)." Then name what makes it slow. "The inner loop recomputes X every time." Once you name the bottleneck — you know what to eliminate. The optimisation targets the bottleneck, always.
Q2
What information am I throwing away that I shouldn't?
Brute force recomputes things. What is being recomputed? Prefix sums cache cumulative totals. Monotonic stacks cache the last relevant element. DP caches subproblem results. If you find what's being thrown away — the data structure to preserve it follows immediately.
Q3
What does optimal NOT do that brute force does?
Brute force checks everything. Optimal skips something safely. What can be skipped and why? "Optimal doesn't restart the window from scratch — it slides." "Optimal doesn't re-examine popped elements — they're permanently dominated." This question gives you the invariant of your algorithm.
Q4
Trace a tiny example. 3-4 elements only.
Not the given example — a simpler one you construct. Run brute force on it by hand. Watch what happens. The pattern almost always becomes visible at small scale before it's visible abstractly. This is the most reliable insight generator. When reasoning fails — always trace.
Q5
If I fix one variable, what's optimal for the rest?
Fix left pointer — what's the best right? Fix index i as peak — what's the best extension? Fix mid as answer — is it feasible? This question breaks two-variable problems open. Most Hard problems have two interacting variables — fixing one makes the other trivially solvable.
Q6
What can be permanently discarded and why?
This is the stack/queue/heap question. If some element can never contribute to any future answer — remove it. Defining exactly what "can never contribute" means for THIS problem gives you the monotonic condition, the DP transition, or the greedy choice. It's different every time — which is why you must derive it, not remember it.
If still stuck after 5 questions — structured unstuck protocol
1
Relax a constraint
What if the array were sorted? What if n=5? Solve the easier version. The easier version almost always reveals the algorithm for the real version.
2
Invert the problem
"What would make this impossible?" "What does the worst case look like?" "What if I worked backwards?" Stamping The Sequence, Jump Game II, Candy — all solved cleanly by inversion.
3
State what you know in comments
"I know I need X. I know Y is true. I'm stuck on Z." Writing the gap explicitly often solves it. Your brain processes named problems faster than unnamed ones.
Phase 4 — Code (10-15 min for Medium, 15-20 for Hard)
1
Map insight to 3 things before typing
Insight → data structure + invariant
Insight → loop direction + number of passes
Insight → result variable + update condition

Example — sliding window:
Data structure: two pointers + running sum
Loop: left to right, single pass
Result: max_len, updated when window is valid
2
Write skeleton first — no logic yet
int solve(vector<int>& nums, int k) {
  int left = 0, result = 0;
  for (int right = 0; right < n; right++) {
    // expand window
    // shrink window
    // update result
  }
  return result;
}
Each comment becomes exactly one block. Fill them one at a time. Never jump between sections.
3
Name every variable with its invariant
"left = start of valid window. sum = sum of current window. max_len = best answer seen so far." If you can't state what a variable represents at every point in the loop — rename it or rethink it. Vague variable names cause vague bugs.
4
Fill logic one concern at a time
Complete "expand window" fully. Then "shrink window." Then "update result." Never write logic for the next section while the current one is incomplete. Context switching mid-section is where off-by-one errors are born.
5
Watch for these bugs before submitting
Integer division when result should be double — always cast explicitly
Off-by-one in loop bounds — check first and last iteration manually
Wrong strict vs non-strict in comparisons — causes double counting
Result initialised to 0 when answer could be negative — use INT_MIN
Single pass when two passes needed — mixing left and right information
Stack storing values when indices needed — almost always store indices
Phase 5 — Test (5 min)
1
Mental dry run on given example first
Trace through your actual code — not your mental model of it. Line by line. Catch the gap between what you think the code does and what it actually does. This alone catches 50% of bugs before submission.
2
Run the 12 universal cases
Empty input []
Single element [x]
All same [x,x,x]
All negative [-3,-1,-5]
Sorted increasing
Sorted decreasing
Answer = whole array
Answer = single element
Duplicates at boundaries
n=2 (smallest valid)
Target at index 0
Target at index n-1
3
Generate your own edge cases with 3 questions
What does my code assume about input? Create a case that violates each assumption.
What happens at the first and last iteration of my loop?
What is the smallest valid input — n=0, n=1, n=2?
4
State complexity before submitting
"Time: O(n log n). Space: O(n)." If you can't state it — you don't fully understand your solution. Don't submit until you can say it out loud.
When stuck — exact protocol by time
0-10 min stuck
Trace a tiny example by hand
3-4 elements. Run brute force manually. Watch what happens. Most insights appear immediately at small scale. When reasoning fails — always trace first.
10-20 min stuck
Run all 5 insight questions on paper
Write answers to Q1-Q5 from Phase 3. Don't skip any. The insight is almost always the answer to Q2, Q3, or Q5. The act of writing forces your brain to reason structurally instead of searching by template.
20-30 min stuck
Relax constraints + invert
Solve easier version first. What if array were sorted? What if n=3? What if you worked backwards? The easier version reveals the real algorithm 80% of the time.
30+ min stuck
Read only the first sentence of editorial
Not the solution. Not the code. Just the first sentence. Then close it and try again for 10 more minutes. This gives you the structural hint without removing the derivation work — which is where the learning lives.
